### Anthropic Engineering: Scaling Managed Agents — Decoupling the brain from the hands

Anthropic Engineering has published a deep technical post on their new Managed Agents architecture.
Anthropic Engineering 发布了一篇关于新托管 Agent 架构的深度技术文章。

The core idea: decouple the "brain" (Claude + harness) from the "hands" (sandboxes and tools) and from the "session" (the durable event log).
核心思路：将"大脑"（Claude + harness）与"双手"（沙盒和工具）以及"会话"（持久事件日志）彻底解耦。

In the old coupled design, everything ran in one container — if it died, the session was lost, and credentials could be exposed to prompt injection from the sandbox.
旧的耦合设计中，所有组件运行在同一容器内——容器崩溃则会话丢失，且沙盒内的提示注入攻击可能暴露凭证。

By separating concerns, the harness is now stateless: a crashed harness can be rebooted with wake(sessionId), and it resumes from the last event in the log.
通过关注点分离，harness 现在是无状态的：崩溃的 harness 可通过 wake(sessionId) 重启，从日志中最后一条事件恢复执行。

The security improvement is structural: credentials never land in the sandbox where Claude's generated code runs.
安全性改进是结构性的：凭证永远不会进入 Claude 生成代码的运行沙盒。

Git tokens are wired into the local remote at clone time; OAuth tokens live in a vault external to the sandbox, accessed through a proxy.
Git token 在克隆时写入本地 remote；OAuth token 保存在沙盒外的 vault，通过代理访问。

The performance payoff was dramatic: p50 time-to-first-token (TTFT) dropped ~60%, and p95 dropped over 90%, because containers are now provisioned on demand via tool call rather than up front.
性能提升显著：p50 首 token 延迟（TTFT）下降约 60%，p95 下降超过 90%，因为容器现在通过工具调用按需分配，而非预先启动。

The session log also replaces the context window for long-horizon tasks — getEvents() lets the harness selectively fetch, rewind, or reread prior context, keeping irreversible compaction decisions in the harness rather than baked in.
会话日志还取代了长时任务中的上下文窗口——getEvents() 让 harness 可以选择性地获取、回溯或重读历史上下文，将不可逆的压缩决策保留在 harness 中而非硬编码。

The framing borrows from OS design: "how to design a system for programs as yet unthought of" — by virtualizing agent components into stable interfaces (session, harness, sandbox) so that implementations can be swapped as models improve.
框架借鉴了操作系统设计理念："如何为尚未被构想的程序设计系统"——通过将 agent 组件虚拟化为稳定接口（session、harness、sandbox），使实现层可随模型进步自由替换。

Source: https://www.anthropic.com/engineering/managed-agents

### No Priors — SAP: Bringing the 'Operating System' of a Company into the AI Era with CTO Philipp Herzig

SAP CTO Philipp Herzig argues that AI is fundamentally a business model transition, not just a technology transition.
SAP CTO Philipp Herzig 认为，AI 从根本上是商业模式的转型，而不仅仅是技术转型。

SAP serves 400,000 enterprise customers and describes itself as the "operating system" of a company, covering finance, HR, supply chain, manufacturing, and sales end-to-end.
SAP 服务 40 万家企业客户，将自身定位为公司的"操作系统"，端到端覆盖财务、人力资源、供应链、制造和销售。

Herzig identifies three layers being transformed by AI: the UI (moving to generative, dynamic UIs rather than human-taught click-through interfaces), business processes (blending structured and unstructured data via agents), and the data layer (harmonizing fragmented enterprise data to fuel AI).
Herzig 识别出 AI 正在变革的三个层面：UI（从人类学习点击的界面转向生成式动态 UI）、业务流程（通过 agent 融合结构化与非结构化数据），以及数据层（统一碎片化的企业数据以驱动 AI）。

The biggest engineering challenge is not AI itself, but "teaching the AI to do the right thing at scale" — he uses the MCP example: 10 APIs are easy, 100 causes context bloat, SAP has 20,000.
最大的工程挑战不是 AI 本身，而是"教会 AI 在规模上做正确的事"——他以 MCP 为例：10 个 API 很简单，100 个就会导致上下文膨胀，而 SAP 有 20,000 个。

He made a strong case for evals, drawing a historical analogy to test-driven development: the reason TDD never caught on was that it was "more fun to write the code first" and requirements were messy — but now that code generation is automated, defining the correct outcome becomes the core skill.
他为 eval 进行了有力论证，以测试驱动开发做历史类比：TDD 之所以未能普及，是因为"先写代码更有趣"且需求混乱——但现在代码生成已自动化，定义正确结果成为核心技能。

SAP published research on Relational Pretrained Transformers (RPT-1): a transformer-based architecture specifically for tabular/structured data — enabling accurate predictions (demand forecasting, cash flow, payment delays) without large training datasets, bringing "the same democratization LLMs gave the unstructured world" to the structured world.
SAP 发布了关系预训练 Transformer（RPT-1）的研究：一种专为表格/结构化数据设计的基于 transformer 的架构——无需大量训练数据即可实现精准预测（需求预测、现金流、付款延迟），将"LLM 带给非结构化世界的民主化"引入结构化世界。

On pricing: SAP is moving from seat-based to a hybrid consumptive model, with outcome-based licensing as the eventual direction — but enterprise customers still demand predictability, so the transition is gradual.
关于定价：SAP 正从基于席位向混合消耗型模式转型，以结果型许可证为最终方向——但企业客户仍需要可预测性，因此转型是渐进式的。

On what stays durable: "Our job at SAP is to make the technology disappear. We need to get the outcome in front of the customer."
关于持久性的核心：" SAP 的工作是让技术消失。我们需要将结果呈现在客户面前。"

He is also exploring quantum computing for optimization problems (traveling salesman, route planning, logistics) — not because the hardware is ready, but to develop the algorithms early so SAP can be hardware-agnostic when it matures.
他还在探索量子计算在优化问题上的应用（旅行商问题、路线规划、物流）——不是因为硬件已就绪，而是为了提前开发算法，以便在硬件成熟时 SAP 能保持硬件无关性。
